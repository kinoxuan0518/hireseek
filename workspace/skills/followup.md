# 跟进未回复候选人（Followup）

版本：1.0
执行方式：浏览器自动化 + 历史数据

## 0. 执行边界

- 仅处理状态为 `contacted`（已联系但未回复）的候选人。
- 同一候选人最多跟进 2 次，超过后标记为 `on_hold`，不再打扰。
- 跟进消息必须与首次消息有明显区别，不能直接重发。

## 1. 启动前检查

1. 从 SQLite 数据库读取符合条件的候选人列表：
   - 状态为 `contacted`
   - 首次联系时间在 2-7 天前
   - 跟进次数 < 2
2. 按渠道分组（BOSS / 脉脉 / LinkedIn）。
3. 确认各渠道均已登录。

## 2. 跟进策略

### 2.1 第一次跟进（首次联系后 3-5 天）

话术方向：
- 换个角度介绍岗位
- 提及最新进展或团队亮点
- 语气轻松，给对方台阶下

示例：
```
你好，上次发了消息，不知道是不是没看到～
[补充一个新的岗位亮点或团队信息]，
有兴趣的话随时聊！
```

### 2.2 第二次跟进（首次联系后 7-10 天，若无回复）

话术方向：
- 简短，不强求
- 表明机会仍在
- 给出明确的结束暗示

示例：
```
最后跟进一下，[岗位] 还有在推进，
时机不合适完全理解，有需要随时找我～
```

## 3. 执行流程

### 对每位候选人：

1. 进入对应渠道的聊天界面
2. 找到与该候选人的对话
3. 确认对方确实未回复（非误判）
4. 生成个性化跟进消息（基于候选人背景 + 上次消息内容）
5. 发送消息
6. 更新数据库：
   - 记录跟进时间
   - 跟进次数 +1
   - 若第二次跟进完成，状态改为 `on_hold`

## 4. 终止条件

- 所有待跟进候选人处理完毕
- 用户主动结束
- 渠道风控触发（立即停止）

## 5. 执行报告

```
跟进完成报告
─────────────
处理候选人：N 人
  - 成功发送：X 人
  - 已达跟进上限：Y 人（标记 on_hold）
  - 跳过（已回复/不符合）：Z 人

渠道分布：
  BOSS直聘：X 人
  脉脉：X 人
  LinkedIn：X 人
```

## 6. 异常处理

| 异常 | 处理方式 |
|------|----------|
| 找不到候选人对话 | 记录日志，跳过该候选人 |
| 对方已回复（数据库未同步）| 更新状态为 replied，跳过 |
| 渠道限制 | 停止该渠道，继续其他渠道 |
