# BOSS直聘消息与简历处理技能

## 目标

- 清空待处理消息并推进候选人状态。
- 使用已存储岗位规则做一致评估，不临时改标准。
- 执行“先判定再下载”，减少无效下载和无效上传。
- 默认执行顺序固定为：先处理消息，再处理今天简历；满足触发条件时自动上传飞书并提交闭环。
- 批量导入飞书时稳定处理三种状态：正常、缺必填项、重复人才。
- 将本轮结构化结果可选同步到飞书多维表格，形成可追踪的数据闭环。

## 脱敏约定（公开仓库）

- 文档中只使用占位符，不包含个人账号、个人路径、企业内部 URL、App 凭据。
- 本地目录统一使用变量：`<DOWNLOAD_DIR>`、`<QUARANTINE_DIR>`、`<CACHE_FILE>`。
- 飞书相关配置统一使用变量：`<APP_ID>`、`<APP_SECRET>`、`<APP_TOKEN>`、`<TABLE_ID_*>`。
- 禁止提交任何真实密钥到仓库；实际值仅在本地 `.env` 或密钥管理系统配置。

## 强规则（必须执行）

1. 默认只处理“今天收到/今天需处理”的简历，除非用户明确放宽日期范围。
2. 规则读取优先级固定为：`effective_rules > jobs[].draft_overrides > global_rule_defaults`。
3. 默认顺序固定为：`先处理消息 -> 再处理今天简历 -> 条件满足则上传飞书并提交`。
4. 必须先判定是否通过岗位规则，再下载简历。
5. 下载硬闸：仅当候选人状态为 `PRECHECK_PASSED` 或 `PRECHECK_OVERRIDDEN` 且终判完成时，才允许下载。
6. 在 BOSS 不通过操作中，统一选择 `其他原因`，避免触发额外消息。
7. 评估标准优先读取岗位生效规则（effective rules），不得擅自放宽或收紧硬性条件。
8. 本科院校要求为硬性条件（以岗位缓存生效规则为准）；硕士/博士背景不能豁免本科不符。
9. 硬性条件（学历/年限/核心技能）任一不通过即判定不通过；仅在用户明确特批时可保留。
10. 用户特批保留时，必须写入缓存：`override_flag=true` + `override_reason`。
11. 进入飞书批量导入后，先处理 `缺必填项`，再处理 `重复人才`，最后提交。
12. 飞书页面操作前先清除遮挡弹窗（反馈/引导/浮层），再点击核心按钮。
13. 下载策略默认“直接落盘到目标目录”；仅在浏览器限制下使用临时目录中转并立即迁移。
14. 禁止将“打印/打印预览”作为常规简历处理界面；合适简历应在会话主界面直接下载。
15. 处理消息时必须先在 `沟通中/全部` 视角清点总消息，再按 `已获取简历` 等细分视角处理。
16. 处理中若未读消息数相对本轮起点上升，必须中断当前队列并优先处理新增未读。
17. 未打开简历内容并提取证据前，不得做最终 `不合适` 结论（纯硬性条件命中除外）。
18. 不通过结论必须写入结构化拒绝原因码 `reject_reason_code`；BOSS 端操作仍统一选择 `其他原因`。
19. 飞书上传前必须做“本科硬闸二次校验”：仅 `undergraduate_passed=true` 的候选人可进入上传名单。
20. 本科信息缺失、院校识别不确定或别名无法归一时，按硬拒处理（`SCHOOL_EVIDENCE_MISSING`）。
21. 飞书批量导入若用户指定统一职位，必须对本批全部简历使用该职位。
22. 自动上传飞书仅在“本轮新增通过人数 > 0 且已下载待上传数 > 0”时触发。
23. 截止标准固定为：消息处理完成 + 今日简历处理完成 +（若触发上传）飞书批量提交成功并返回候选人列表。
24. 若启用飞书多维表格同步，必须在“提交飞书招聘后”执行数据写入，禁止提前写入“已上传”状态。
25. 多维表格写入必须幂等：以 `platform_id + job_id` 或 `interaction_id` 为唯一键 upsert。
26. 多维表格写入失败不得影响主流程终态，但必须记录 `ERROR_BITABLE_SYNC` 并输出重试指引。

## 输入前置

执行前确认：

- 已登录 BOSS 直聘企业端。
- 已登录飞书招聘。
- 已知当前职位及其生效规则（来自岗位缓存/用户确认规则）。
- 飞书批量导入来源为 `第三方招聘网站 -> BOSS直聘`。
- 若用户指定“统一加入职位”，已确认职位名（示例：`AI 研究开发工程师`）。
- 已配置本地路径：
  - `<DOWNLOAD_DIR>`：通过候选人简历下载目录。
  - `<QUARANTINE_DIR>`：误下载隔离目录。
  - `<CACHE_FILE>`：幂等缓存文件。
- 若启用多维表格数据层，已在本地配置：
  - `<APP_ID>`、`<APP_SECRET>`、`<APP_TOKEN>`。
  - `<TABLE_ID_CANDIDATES>`、`<TABLE_ID_INTERACTIONS>`、`<TABLE_ID_DASHBOARD>`。
- 默认请求简历话术可用：

```text
可以的话，要不发我一份简历，我们根据简历来简单聊聊？
```

## 执行前清障（必做）

进入飞书批量新建页面后，先执行一次清障：

1. 关闭右下角 `反馈` 弹窗（优先点 `取消`，不提交反馈）。
2. 关闭新手引导、客服浮层、遮挡上传区域的悬浮窗。
3. 若遮挡未消失，按 `Esc` 后重试点击目标按钮。
4. 仍无法点击时记录 `ERROR_UI_BLOCKED`，刷新页面后重试一次。

## 触发命令

| 命令 | 动作 |
|---|---|
| `处理消息和简历` | 默认闭环：先处理消息，再处理今天简历；满足触发条件自动上传飞书并提交 |
| `处理今天简历` | 只处理今天范围内候选人；按“先判定再下载”执行 |
| `处理消息` | 扫描会话并请求/接收简历 |
| `处理简历` | 按岗位规则完成评估并给出通过结论 |
| `批量上传飞书` | 执行飞书批量新建并处理缺项/重复 |
| `批量处理简历` | 从 BOSS 到飞书整批闭环处理 |
| `同步飞书多维表格` | 将本轮结构化结果 upsert 到飞书多维表格 |

## 执行原则

1. 先识别状态，再执行动作。
2. 每位候选人一次只推进一个状态。
3. 任何会触达候选人的动作都遵循最小触达原则。
4. 使用 `candidate_id + job_id` 幂等键，已完成步骤默认跳过。
5. 批量模式每处理 5 人输出一次进度。
6. 每次点击前确认页面上下文（当前 Tab、当前弹窗、当前候选人）与目标动作一致。
7. 默认闭环执行，不把“上传飞书”作为可遗漏的人工补充步骤。
8. 若启用 Bitable，同步步骤作为闭环最后一步执行（不阻塞主流程终态）。

## 状态机

```text
NEW_MESSAGE
  -> REQUESTED_RESUME
  -> RESUME_RECEIVED
  -> PRECHECK_PASSED / PRECHECK_REJECTED / PRECHECK_OVERRIDDEN
  -> DOWNLOADED
  -> FEISHU_IMPORTING
  -> FEISHU_FIXED_REQUIRED_FIELDS
  -> FEISHU_DUPLICATE_UPDATED
  -> FEISHU_SUBMITTED
  -> BITABLE_SYNCED
  -> DONE

任意状态可转：REJECTED / ON_HOLD / ERROR
```

## 主流程

默认流程顺序：
`消息处理 -> 简历评估 -> 下载 -> (满足触发条件时)飞书批量上传 -> 飞书提交 -> (可选)Bitable 同步`

### 1) BOSS 消息扫描与分类

进入消息列表后，按以下顺序收集待处理会话：

- 先切换到 `沟通中/全部` 视角确认未处理消息总量，再进入细分视角执行。
- 记录本轮基线未读数 `unread_baseline`，后续每处理 1 个会话后复核一次。
- 候选人已回复但我方未回复。
- 候选人主动投递并等待处理。
- 候选人发起“发送简历”请求待同意。
- 会话中已有简历但尚未评估。
- 若发现 `unread_now > unread_baseline`，立即中断当前队列，优先处理新增未读。

### 2) 会话处理

对每个会话执行：

1. 打开会话并读取最近消息。
2. 判断分支：
- 已有简历：跳转到“简历评估”。
- 无简历且候选人有意向：发送标准请求简历话术。
- 出现“对方想发送简历给你”：点击“同意”。
- 候选人明确拒绝或不匹配：标记 `REJECTED` 并结束该会话。
3. 写入缓存并返回消息列表。
4. 返回列表后检查未读是否上升；若上升，切换到新增未读优先队列。

### 3) 先判定再下载（关键）

在 BOSS 端先完成预判：

- 按优先级读取规则：`effective_rules > jobs[].draft_overrides > global_rule_defaults`。
- 阶段 A：硬性条件预判。
- 阶段 B：打开简历内容做证据复核后给最终结论。
- 命中硬性不通过：`PRECHECK_REJECTED`，不下载，进入不通过处理。
- 用户明确特批保留：`PRECHECK_OVERRIDDEN`，写入特批原因后允许下载。
- 通过：`PRECHECK_PASSED`，允许下载并进入上传候选池。

### 4) 简历评估规则

按岗位规则输出 100 分制结果，先硬性条件，后加权评分。

#### 4.1 硬性条件（必须先判）

- 学历门槛：本科院校必须满足岗位生效规则（示例：`985/QS100`）。
- 工作年限门槛。
- 核心技能门槛（示例：LLM/RAG/PyTorch）。

任一不通过即结论 `不合适`。

#### 4.2 权重评分（总分 100）

- 教育背景：25
- 工作经历与公司背景：25
- 核心技能匹配：30
- 领域经验与项目产出：20

#### 4.3 输出模板

```text
简历评估
候选人：<姓名>
职位：<职位名称>
总分：<0-100>
结论：<合适/不合适/待定>

硬性条件：
- 学历：<通过/不通过>
- 年限：<通过/不通过>
- 核心技能：<通过/不通过>

关键证据：
- <证据1>
- <证据2>

风险点：
- <风险1>

建议动作：<上传飞书/保留观察/淘汰>
请用户确认：合适 / 不合适 / 待定
```

### 5) 不通过处理（BOSS）

当结论为不通过：

1. 若当前在简历预览层，先返回会话主面板再执行动作。
2. 点击 `不合适`，理由统一选择 `其他原因`。
3. 写入 `reject_reason_code`，不发送额外说明消息。
4. 状态写入 `REJECTED`。

### 6) 下载与整理

仅对 `PRECHECK_PASSED` 或 `PRECHECK_OVERRIDDEN` 的候选人下载简历：

1. 下载前执行硬闸校验。
2. 在会话主界面直接下载，禁止以打印页作为常规下载入口。
3. 优先直接下载到 `<DOWNLOAD_DIR>`。
4. 若浏览器只能落到默认目录，下载后立即迁移到 `<DOWNLOAD_DIR>` 并校验存在。
5. 校验失败标记 `ERROR_DOWNLOAD_MISSING`。
6. 误下载立即迁移到 `<QUARANTINE_DIR>` 并标记 `ERROR_DOWNLOAD_POLICY_BREACH`。
7. 文件名保留原始姓名信息；重名时追加短后缀（如 `_2`）。

### 7) 飞书批量上传（招聘系统）

1. 仅当满足“新增通过人数 > 0 且已下载待上传 > 0”时自动触发。
2. 点击 `+新建 -> 批量新建人才`。
3. 来源选择 `第三方招聘网站 -> BOSS直聘`。
4. 若用户指定统一职位，`加入职位` 统一选择该职位。
5. 生成 `upload_whitelist`：仅 `PRECHECK_PASSED/PRECHECK_OVERRIDDEN` 且 `undergraduate_passed=true`。
6. 上传前复核 `upload_whitelist` 的本科证据；不确定即剔除。
7. 仅上传白名单内简历，禁止按目录全量上传。
8. 出现“成功上传 N 份简历”后进入下一步。

### 8) 飞书逐条处理（三种状态）

#### 8.1 正常简历

- 不修改，保持待提交。

#### 8.2 缺必填项

1. 点击该条 `缺必填项`。
2. 在右侧表单定位红色校验提示。
3. 优先修复教育经历起止时间；结束时间缺失且在读时点击 `目前年月`。
4. 仍缺失则手动补齐 `YYYY-MM`。
5. 确认该条 `缺必填项` 标签消失。

#### 8.3 重复人才

1. 点击 `重复人才`。
2. 进入 `发现重复人才` 核对身份信息。
3. 至少两项关键信息一致才按重复更新。
4. 冲突则标记 `ERROR_DUPLICATE_CONFLICT` 并置 `ON_HOLD`。
5. 确认弹窗显示重复更新数量后点击 `确定`。

### 9) 提交与完成

当满足以下条件再提交：

- `缺必填项=0`
- 重复人才已核对且无 `ERROR_DUPLICATE_CONFLICT`
- 上传成功数 = 待上传数

执行：

1. 点击 `提交`。
2. 确认弹窗点击 `确定`。
3. 返回候选人列表后标记 `FEISHU_SUBMITTED`。

### 10) 飞书多维表格同步（可选但推荐）

若启用 Bitable 数据层：

1. 读取本轮结构化结果（候选人、交互、职位统计）。
2. 候选人主档：按 `platform_id` upsert。
3. 交互记录：按 `interaction_id` 去重写入。
4. 职位看板：按 `job_name` upsert 漏斗指标。
5. 成功后标记 `BITABLE_SYNCED`；失败标记 `ERROR_BITABLE_SYNC` 并记录重试指令。

详细配置与 API 示例见：
- `references/feishu_bitable_llm_guide.md`

## 批量模式

启动命令：`批量处理简历`。

- 优先级：今天已获取简历 > 今天待同意收简历 > 今天待请求简历。
- 节奏：逐个处理，不并发点击。
- 汇报：每 5 人输出一次进度；结束后输出总汇总。
- 中止条件：
  - 连续 3 次页面异常或按钮不可点击。
  - 飞书页面连续 2 次无响应。
  - 用户发送“停止”。

## 缓存与幂等

缓存文件：`<CACHE_FILE>`

建议结构：

```json
{
  "schema_version": "1.3",
  "updated_at": "2026-03-04T00:00:00Z",
  "candidates": {
    "<candidate_id>#<job_id>": {
      "status": "EVALUATED",
      "score": 82,
      "decision": "ON_HOLD",
      "decision_source": "effective_rules",
      "override_flag": false,
      "override_reason": "",
      "undergraduate_passed": true,
      "reject_reason_code": "",
      "last_action": "resume_review",
      "last_updated": "2026-03-04T00:00:00Z"
    }
  }
}
```

规则：

- 写入前先读缓存，命中幂等键且目标状态已完成则跳过。
- 状态回退只能在用户明确指令下执行。
- 用户特批通过必须落缓存：`decision_source=manual_override`、`override_flag=true`、`override_reason` 不为空。

## 异常处理

- 简历无法识别：提示用户补充关键信息后继续。
- 飞书页面无响应：刷新后重试 2 次，仍失败标记 `ERROR`。
- 职位列表为空：提示检查飞书权限或职位配置，不盲目上传。
- 上传失败：记录错误并提供“重试上传”单步动作。
- 点击被遮挡：先清障再重试 1 次，仍失败标记 `ERROR_UI_BLOCKED`。
- 下载后目标目录无文件：执行默认下载目录搜寻与迁移；仍缺失则 `ERROR_DOWNLOAD_MISSING`。
- Bitable 同步失败：记录 `ERROR_BITABLE_SYNC`，输出待补偿数据包，不影响本轮主流程完成标记。

## 与 bosszhibin-auto-recruiter 协作边界

- `bosszhibin-auto-recruiter`：职位解析、规则缓存、主动打招呼与初筛。
- 本技能：消息收口、严格判定、下载、飞书批量导入与修复提交、可选 Bitable 同步。
- 交接信号：候选人进入“可收简历/已收简历/已获取简历”状态后切换到本技能。
